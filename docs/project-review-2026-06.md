# Dopamine — Project Review (June 2026)

A critical review of the Dopamine visual-effects library: the portable `.dope`
format, its open-source posture, its fitness for the AI-coding moment, and the
competitive landscape. Written against the repo as of 2026-06-12 (141 commits,
first commit 2026-06-08). Every internal claim below is checkable against the
tree; external claims carry URLs.

**Scope note on candor:** this review names weaknesses plainly. The project is
four days old; several of the problems below are simply "young project"
problems. They are listed anyway, because the gap between the README's ambition
and the repo's shipping state is the single most important thing to manage.

---

## 0. Executive summary

- **The engineering thesis is sound and rare.** "One portable data file, three
  thin interpreters, parity gated by byte-identical fixtures" is executed with
  unusual rigor: a 192-case cross-platform parity grid, single-source
  GLSL→MSL/Kotlin shader generation with golden-snapshot and SwiftShader
  RGB-Δ0 gates, and a binding contract that drives both runtime binding and
  Metal codegen. Six of ten effects ship with zero platform-specific code.
- **The project is not yet open source in any meaningful sense.** No LICENSE
  file, nothing published to npm/SwiftPM/Maven (all packages `0.1.0`, root
  `private`), the npm name `dopamine` is squatted, the full CI gate requires a
  paid macOS runner that forks can't use, and the contributor base is one
  human plus an AI agent (31 + 110 of 141 commits).
- **The format's biggest weakness is ergonomic, not semantic.** Expression
  trees as raw JSON (fail's amp: 60 lines for ~4 lines of math) make humans
  unable to author effects unassisted, and there is no `validate` command, no
  surface syntax, and no authoring tool. The semantics — a deliberately tiny
  grammar — are defensible; the authoring surface is not, yet.
- **Strategically, the AI-coding moment is the opening.** A schema-validated,
  machine-checkable, render-verifiable effect format is the ideal target for
  LLM generation, and the repo itself is proof (it was mostly agent-built).
  Competitors are retrofitting AI workflows onto formats with no determinism
  or verification story; Dopamine has the verification story and no workflow
  packaging. That asymmetry is the window — and windows close.
- **Nobody else occupies the niche** (portable, deterministic, GPU-shader
  reward moments across web/iOS/Android), but the niche is small and Lottie is
  "good enough" for buyers who don't care about shader-grade visuals. Winning
  requires distribution work (publish, demo, compat shims, agent packaging)
  that the repo has none of today.

## 1. The format: usefulness and expressiveness

### 1.1 What the format actually is

One `.dope.json` per effect (`effects/<name>/<name>.dope.json`) declaring:

- a **mood→params mapping grammar** (`lerp`/`mul`/`baseline`/`control`/
  `round`/`clamp`/`if`/`step`/… — see `docs/effect-format.schema.json`),
  evaluated with a deterministic PRNG so `mood × intensity × whimsy × seed`
  resolves to byte-identical numbers on web, Swift, and Android;
- a **per-frame expression grammar** (`tempo.frame`) with 16 node types
  (`add clamp const div easeOutBack easeOutCubic envelope exp lt max min mul
  param pow sin sub` — `packages/core/src/framework/frame-expr.ts`);
- a **binding contract** (which params are uniforms, scatter, samplers,
  per-frame extras, uniform-array geometry) that both the runtimes and the
  codegen consume;
- `x-build` blocks from which the toolchain generates the MSL and Kotlin
  shaders, the Swift/Kotlin factories, and (for lightning) a transpiled
  CPU-geometry renderer.

The spec is substantial and real: `docs/effect-format.md` is 937 lines,
`docs/effect-format.schema.json` 459 lines, and parity is enforced by a
192-case fixture grid run natively on all three stacks
(`swift/Tests/DopamineCoreTests/ParityTests.swift`,
`android/.../ParityTest.kt`, regenerated by `swift/Scripts/regen-parity.sh`).

### 1.2 What it gets right

| Strength | Evidence |
|---|---|
| **Parity by fixture, not by trust** | 192-case grid asserted byte-identical on web/Swift-Linux/pure-JVM; `.dope` embeds md5-checked in CI (`android.yml` jvm job) |
| **One source of truth for uniforms** | The `binding` contract ships in the portable file *and* drives Metal struct codegen — no per-platform drift surface |
| **Single-source shaders** | GLSL ES 3.00 authored once; MSL + Kotlin generated (`tools/dopamine/src/shader.mjs`, `android-shader.mjs`), gated byte-for-byte by golden snapshots and a SwiftShader RGB-Δ0 mid-frame gate (`scripts/shader-goldens.mjs`) |
| **Declarative effects ship zero platform code** | 6 of 10 effects (aurora, fail, halo, inkstroke, lightning, ripple) have no `swift/` or `android/` folder at all — factories generated from the `.dope` |
| **Honest escape hatches** | Panel/hybrid effects (comic, confetti, heartburst, solarbloom) keep hand-written platform sources; `hooks` and `x-build.logic` cover code-shaped pieces without polluting the core |
| **A real loop contract** | `tempo.loop` (§7.2) validates seam invariants at parse time rather than hoping authors get period math right |

The deepest idea here — *the portable data file matters more than the code; the
code on every platform is an interpreter* — is the right idea, and the test
architecture takes it seriously. Very few animation projects gate cross-platform
determinism this hard. This is the project's genuine moat.

### 1.3 Where expressiveness falls short

**Expression trees are hostile to humans.** `effects/fail/fail.dope.json`'s
`tempo.frame.amp` is 60 lines of nested JSON to express what would be ~4 lines
of infix math. The shape of the problem, schematically:

```jsonc
// what the .dope requires (fragment of the real 60-line tree):
{ "mul": [ { "envelope": [ { "param": "life" }, { "const": 0.35 } ] },
           { "add": [ { "const": 0.6 },
                      { "mul": [ { "const": 0.4 },
                                 { "sin": { "mul": [ /* … */ ] } } ] } ] } ] }
// what an author thinks:
// amp = envelope(life, 0.35) * (0.6 + 0.4 * sin(…))
```

There is no textual surface syntax, no authoring tool, and no
`dopamine validate`/`explain` command — the CLI accepts exactly one subcommand,
`build` (`tools/dopamine/src/cli.mjs:77-78`). Today the only practical authors
are the toolchain's own developers and LLMs (see §3 — this is double-edged).

**The grammar is small by design, and the boundary will be tested.** No
user-defined functions, no vectors (scalars only), no `smoothstep`/`atan`/
`noise` in frame-expr, no conditional richer than `lt`/`if`/`step`. The repo's
own history shows the pressure valve: lightning's geometry couldn't fit, so
`x-build.logic` (a restricted-TS transpiler) was added. That is a *second*
language with a *second* parity mechanism. Each future effect that doesn't fit
will tempt another mechanism; the per-effect seams are good, but the format's
"what fits in pure data" boundary needs to be policed deliberately, or the
spec's surface area will grow faster than its three interpreters can be kept
honest.

**The interaction model is thin.** One-shot envelope or seamless loop — that's
the entire temporal vocabulary. No state machines, no input-driven transitions,
no events, no data binding. Competing formats (dotLottie v2, Rive) made
interactivity the centerpiece of their 2025–26 releases (§4). Dopamine's
mood/intensity/whimsy knobs are a charming and *differentiated* parameter
space — "emotional parameters" rather than timelines — but they resolve once at
`prepare()`; nothing reacts mid-play.

**Content is GPU-only.** Effects are full-screen shader passes (plus panel
textures). There is no vector/path layer, so the format cannot represent the
icon-and-illustration animations that dominate actual product usage of
Lottie/Rive. That's a legitimate scoping choice — "reward moments, not icons" —
but it should be stated as one, because it bounds the addressable use cases.

**Verdict:** as a *portable shader-effect interchange format with enforced
determinism*, `.dope` is genuinely novel and well-executed. As a general motion
format it is not competitive and should not pretend to be. The expressiveness
problem to fix first is ergonomic (authoring surface), not semantic (more ops).

## 2. Open-source relevance

### 2.1 Hard blockers (today, nobody can legally or practically use this)

| Blocker | Fact |
|---|---|
| **No license** | There is no `LICENSE` file in the repo. Default copyright applies; no one may use, copy, or redistribute. Everything else in this section is moot until this is fixed. |
| **Nothing is published** | Root `package.json` is `"private": true`, all packages at `0.1.0`; no npm package, no SwiftPM tag, no Maven artifact. `dist/` is gitignored, so even the SwiftPM/Gradle "packages" exist only on a builder's machine. |
| **The npm name is taken** | `dopamine` on npm is squatted by an unrelated package. Scoped `@dopaminefx/*` availability is unverified; a rename or scope decision is needed before first publish. |
| **Bus factor ≈ 1** | 141 commits in 4 days: 110 authored by Claude, 31 by one human (`git shortlog -sn`). No CONTRIBUTING.md, no issue templates, no second maintainer. |
| **CI requires a paid runner** | The macOS job needs `macos-15-xlarge` (M2) and a non-zero Actions spending limit (`.github/workflows/swift.yml`, documented in CLAUDE.md). A fork cannot run the full gate suite — contributors can't self-verify the Metal path. |

### 2.2 Softer gaps

- **No versioning story for the format itself.** `.dope` files carry no
  spec-version field discussion in the docs' compatibility terms; three
  interpreters must move in lockstep. Fine in a monorepo; fatal for an
  ecosystem ("effects in other repositories equally supported" is the stated
  goal — those repos will break on every grammar change without a
  format-version + capability negotiation policy).
- **No demo anyone can click.** `npm run dev` and the CI reel videos exist, but
  there's no hosted gallery. For a visual library, the demo *is* the README.
- **Docs are agent-oriented.** CLAUDE.md and `docs/` are excellent for an AI
  working *in* the repo and weak for a developer evaluating *whether to adopt*
  — there is no "install, play one effect in 5 lines" page per platform front
  and center.

### 2.3 Is there an audience?

Yes, a real but narrow one: teams shipping the *same* product on web + iOS +
Android who want reward moments ("delight") without maintaining three effect
implementations. canvas-confetti's enduring popularity (web-only, one effect)
proves demand for the category; nobody currently serves it cross-platform with
GPU shaders. The risk isn't demand, it's that the niche is small enough that
Lottie "good-enough" coverage (a confetti `.lottie` plays on all three
platforms today) satisfies most buyers who don't care about shader-grade
visuals or byte parity.

## 3. Fitness for the AI-powered moment

This is the project's strongest strategic position, and it appears largely
accidental rather than packaged.

**Why it fits:**

1. **Effects are data, and the data is schema-validated.** An LLM emitting a
   new `.dope.json` is emitting a constrained, checkable artifact — far safer
   than emitting three platform codebases. The 60-line expression-tree JSON
   that is miserable for humans (§1.3) is *trivially* writable by a model.
2. **The gates make generation verifiable.** Schema validation, the parity
   grid, golden-snapshot shader transpile, the SwiftShader Δ0 gate, and the
   reel render mean a generated effect can be machine-judged end to end. This
   is exactly the loop agentic tooling needs: generate → validate → render →
   look. Competitors are bolting this on (LottieFiles now markets connecting
   "Claude, Cursor and Windsurf" to its editor, https://lottiefiles.com/ai;
   Motion sells an agent-facing `/motion` skill via Motion+, https://motion.dev/);
   Dopamine's repo already *is* that harness — and was substantially built by
   one (110/141 commits agent-authored).
3. **Single-source everything minimizes the surface an agent can desynchronize.**

**Why it doesn't fit yet:**

- **No standalone `validate`.** The only way to check a `.dope` is to run the
  full build inside the monorepo. An external agent (or a CI bot in another
  repo — the stated "effects in other repositories" goal) needs
  `dopamine validate <file>` and `dopamine render <file> --frame 500` as
  first-class, dependency-light commands. The CLI has neither
  (`tools/dopamine/src/cli.mjs`).
- **No agent packaging.** No MCP server, no published skill, no
  "generate an effect" prompt-pack. Given that text-to-Lottie tools are now a
  visible cottage industry (kin3o generates dotLottie state machines from
  prompts, https://kin3o.com/; vizGPT emits Lottie JSON,
  https://vizgpt.ai/usecases/ai-generate-lottie), "text-to-.dope, with a
  determinism guarantee no Lottie generator can offer" is a marketable wedge
  that currently exists only as latent capability.
- **The human is still in the loop for taste, and there's no fast feedback
  surface for them** — no hot-reloading playground where a generated `.dope`
  can be tweaked live.

The 2026 framing matters: AI-generated UI is mainstream enough to have a name
("vibe design"; see https://muz.li/blog/vibe-design-in-2026-what-ai-generated-ui-means-for-your-work/
and https://uimotionprompts.com/blog/add-animations-to-vibe-coded-apps), and
the consistent critique of vibe-coded apps is flat, generic motion. A library
whose pitch is "your agent adds a *production-grade, cross-platform, motion-
reduced-aware* reward moment with one JSON file" is aimed at exactly that gap.

### 4.1 The three matchups that matter

**Lottie/dotLottie is the incumbent to be *adjacent* to, not to fight.** Its
2026 stack is formidable on paper — an MIT-licensed v2 spec, state machines,
theming, data binding, a unified Rust/WASM renderer with WebGPU backends — but
the transition from the legacy lottie-web/lottie-ios runtimes to the new
dotLottie players is exactly the kind of ecosystem churn that makes teams
re-evaluate. Dopamine should never be pitched as a Lottie replacement (it has
no vector layer and no designer tool); it should be pitched as the thing you
reach for when a `.lottie` confetti looks flat: full-screen, GPU-lit, palette-
aware moments. The most useful thing to *take* from LottieFiles is their spec
discipline: versioned format documents with explicit player-support matrices
(https://dotlottie.io/spec/) — Dopamine has nothing equivalent and needs it
before any external `.dope` exists.

**The platform shader APIs are the real long-term competitor.** AGSL/
`RuntimeShader` (Android 13+) and SwiftUI's `Shader`/`layerEffect` (iOS 17+)
are training a generation of app developers to write small per-view shaders
with zero dependencies — every Compose "liquid glass" tutorial erodes the
build-vs-buy case for a third-party effects library. But they also create
Dopamine's clearest expansion: those developers are hand-porting GLSL↔AGSL↔MSL
today with no tooling. The GLSL→MSL transpiler plus the binding contract is
80% of a "write the shader once, target AGSL and SwiftUI Shader too" story —
arguably a more broadly wanted product than the effects themselves.

**The AI-generation wave is the timing argument.** LottieFiles is wiring
Claude/Cursor/Windsurf into its editor; Motion sells an agent skill;
text-to-Lottie startups are multiplying. All of them generate artifacts that
cannot be machine-verified for cross-platform fidelity. Dopamine's gate suite
is the missing half of that product, already built. If the project does one
strategic thing in 2026, it should be packaging "prompt → validated, rendered,
parity-checked `.dope`" before a better-funded player notices the gap.

**Net read:** nobody occupies "portable, deterministic, GPU-shader reward
moments across web/iOS/Android." Lottie/Rive own designer-authored vector
motion; Motion/GSAP own web interaction animation; confetti libraries own the
single-effect web niche; platform shader APIs own per-platform one-offs.
Dopamine's cell in that matrix is empty — but it is also the smallest cell, and
the format will only matter if distribution (publishing, demo, compat shims,
agent packaging) catches up to the engineering.

## 5. Prioritized recommendations

**P0 — existential, this week**

1. **Add a LICENSE (MIT or Apache-2.0) and decide the npm name/scope.** Without
   these the project is not open source, whatever the README says. Resolve the
   `dopamine` squat by committing to `@dopaminefx/*` (verify scope availability)
   or renaming once, now, before any links exist.
   *Done when:* `LICENSE` at repo root, license fields in every `package.json`
   /podspec-equivalent, name decision recorded in the README.
2. **Publish 0.1.0 of the web packages + tag the SwiftPM/Gradle artifacts, and
   host the demo gallery.** An unpublished visual-effects library has zero
   evangelizable surface. The reel pipeline already produces the marketing
   asset; put it and a live demo behind a URL.
   *Done when:* `npm install` works from a clean machine, a SwiftPM tag and a
   Maven coordinate resolve, and the README's first screen links a live demo.

**P1 — adoption mechanics, this month**

3. **Ship `dopamine validate` (and a minimal `render --frame`) as standalone
   CLI commands**, runnable against a lone `.dope` outside the monorepo. This
   is simultaneously the contributor on-ramp, the external-effects enabler, and
   the agent-tooling primitive. Then package it: an MCP server / published
   skill exposing validate+render+the format docs, so "text-to-.dope" works in
   any agent harness — this is the AI-moment wedge (§3) and it's mostly
   plumbing over existing code.
   *Done when:* `npx @dopaminefx/cli validate my-effect.dope.json` passes/fails
   with actionable errors outside this repo, and an agent with the MCP server
   can go prompt → valid `.dope` → rendered frame without human glue.
4. **Build the canvas-confetti compat shim** (`@dopaminefx/confetti` exposing the
   canvas-confetti API over the GPU implementation) and write the Motion/GSAP
   "trigger a reward moment" recipes. Cheap, high-leverage distribution that
   borrows three existing audiences instead of building one from scratch.
   *Done when:* a canvas-confetti user can swap the import and see the same
   API produce the GPU effect; two recipe pages exist in `docs/examples`.

**P2 — durability**

5. **De-risk the structural fragilities:** (a) add a `formatVersion` +
   compatibility policy to the spec before any external `.dope` exists —
   three interpreters with no version negotiation is an ecosystem time bomb;
   (b) make fork CI viable by marking the paid macOS job optional-but-advisory
   and keeping Linux/JVM/SwiftShader gates as the must-pass tier (they already
   cover parity and shader Δ0); (c) address the bus factor honestly —
   CONTRIBUTING.md, a second human maintainer, and a stated governance
   intention; (d) write the human-ergonomics layer for `tempo.frame` (a tiny
   infix surface syntax that compiles to the existing trees, or at minimum a
   documented "let the agent write it, here's the prompt" workflow), so the
   60-line-amp problem is a feature with a story instead of a wall.

**Explicit non-recommendations:** don't chase Lottie/Rive into vector content
or state machines yet — the differentiator is deterministic GPU moments, and
the interactivity question should be answered *after* there are external users
demanding it; don't add grammar ops speculatively; don't build a GUI editor
before the CLI/agent surface exists.
